# 创建 Issue

用户正在开发过程中想到了一个 bug/功能/改进。快速捕获它，以便他们可以继续工作。

## 你的目标

创建一个完整的 issue，包含：
- 清晰的标题
- 简要说明这是什么
- 当前状态 vs 预期结果
- 需要修改的相关文件
- 风险/注意事项（如适用）
- 适当的类型/优先级/工作量标签

## 如何实现

**提问**以填补空白 - 要简洁，尊重用户的时间。他们正在工作流程中，希望快速捕获这个想法。通常需要：
- 问题/功能是什么
- 当前行为 vs 期望行为
- 类型（bug/功能/改进）和优先级（如果不明显）

保持问题简短。一条包含 2-3 个针对性问题的消息胜过多次来回。

**搜索上下文**仅在有用时：
- 如果是复杂功能，搜索网络以了解最佳实践
- 使用 grep 搜索代码库以找到相关文件
- 注意你发现的任何风险或依赖

**跳过显而易见的内容** - 如果是简单的 bug，不要搜索网络。如果类型/优先级从描述中已经很清楚了，不要询问。

**保持快速** - 总交流时间少于 2 分钟。保持对话式但简洁。获取你需要的信息，创建工单，完成。

## 行为规则

- 保持对话式 - 询问有意义的内容，而不是检查清单
- 默认优先级：normal（普通），工作量：medium（中等）（仅在不清楚时询问）
- 最多 3 个文件在上下文中 - 仅最相关的
- 使用要点而非段落
